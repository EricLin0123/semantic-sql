# Build Prompt: Furniture Natural-Language Query Tool

Build a small, self-contained AI data-retrieval tool. A user asks questions in plain English about a company's furniture inventory; the system translates the question into a **read-only** SQL query, validates it for safety, runs it against a SQLite database, and replies in natural language. When a question is genuinely ambiguous, the system asks the user a clarifying question instead of guessing.

Keep it as simple as possible. Do not add features, auth, ORMs, migrations, Docker, or a custom frontend beyond what is specified below. Prefer clarity over cleverness.

---

## 1. Fixed technical decisions (do not substitute)

- **Language:** Python 3.11+
- **Database:** SQLite, single file, accessed via the stdlib `sqlite3` module. No SQLAlchemy/ORM.
- **LLM provider:** OpenRouter, via the `openai` Python SDK pointed at `https://openrouter.ai/api/v1`.
- **Model:** `nvidia/nemotron-3-ultra-550b-a55b:free`
- **Orchestration:** LangGraph (`StateGraph`).
- **SQL safety:** `sqlglot` for parsing and validation.
- **Chat UI:** the terminal. A plain Python REPL loop reading from stdin and printing to stdout. No web framework, no TUI library.
- **Secrets:** loaded from `.env` via `python-dotenv`. The key is `OPENROUTER_API_KEY`.

## 3. Database

### 3.1 Schema (`db/schema.sql`)

```sql
CREATE TABLE furniture_category (
    id   INTEGER PRIMARY KEY,
    name TEXT NOT NULL UNIQUE          -- e.g. 'table', 'chair', 'couch'
);

CREATE TABLE furniture_item (
    id          INTEGER PRIMARY KEY,
    category_id INTEGER NOT NULL REFERENCES furniture_category(id),
    subtype     TEXT NOT NULL,         -- e.g. 'square table', 'round table'
    quantity    INTEGER NOT NULL,
    location    TEXT,                  -- e.g. 'Warehouse A'
    unit_price  REAL
);
```

## 4. Read-only database access (`pipeline/db.py`)

- Open the connection in true read-only mode so no code path can mutate the DB:
  ```python
  conn = sqlite3.connect("file:db/furniture.db?mode=ro", uri=True)
  ```
- Expose a `run_query(sql: str) -> list[dict]` that sets `conn.row_factory = sqlite3.Row`, executes, and returns a list of plain dicts. Cap results at 500 rows.
- Expose a `get_schema_ddl() -> str` that returns the `CREATE TABLE` statements as text (read `db/schema.sql`), used to prime the SQL-generation prompt.
- This read-only connection is a **second, independent safety layer** beneath the SQL validator. Both must exist.

## 5. SQL safety layer (`pipeline/sql_guard.py`)

Pure function, no LLM. Signature: `validate_sql(sql: str, allowed: dict[str, set[str]]) -> tuple[bool, str]` returning `(is_valid, reason)`. `allowed` maps table name → set of column names.

Checks, in order — fail closed on the first violation:

1. **Single statement.** `sqlglot.parse(sql, read="sqlite")` must yield exactly one statement. Reject `;`-chained input.
2. **SELECT only.** The parsed root must be a `sqlglot.exp.Select`. Reject anything else (INSERT/UPDATE/DELETE/DROP/ALTER/CREATE/PRAGMA/ATTACH/etc.).
3. **Known tables only.** Walk the AST for `exp.Table` nodes; every table name must be in `allowed`. This blocks `sqlite_master` / `sqlite_schema` probing.
4. **Known columns only.** For every `exp.Column` node with an identifiable name, it must belong to one of the allowed tables. Allow `*` and aggregate expressions like `COUNT(*)`, `SUM(quantity)`.
5. **No dangerous functions.** Reject if the AST contains function calls to a denylist (`load_extension`, `readfile`, `writefile`, `edit`, `fts3_tokenizer`, etc.).

Return a clear human-readable `reason` on failure — it is fed back to the LLM for a retry, so it should describe *what* was wrong (e.g. `"only SELECT statements are allowed"`).

Provide the `allowed` dict derived from the schema, e.g.:
```python
ALLOWED = {
    "furniture_category": {"id", "name"},
    "furniture_item": {"id", "category_id", "subtype", "quantity", "location", "unit_price"},
}
```

## 6. LLM wrapper (`pipeline/llm.py`)

- Construct an `openai.OpenAI` client with `base_url="https://openrouter.ai/api/v1"` and `api_key=os.environ["OPENROUTER_API_KEY"]`.
- One helper `chat(system: str, messages: list[dict]) -> str` that calls `client.chat.completions.create(model=MODEL, ...)` and returns the message content string.
- The SQL-generation node needs **structured** output. Instruct the model (in the system prompt below) to return raw JSON only. Parse defensively: strip markdown code fences if present, then `json.loads`. If parsing fails, treat it as a validation failure and retry.

## 7. System prompts (`pipeline/prompts.py`) — use verbatim

### 7.1 `SQL_GENERATION_PROMPT`

```
You are a SQL generation assistant for a furniture inventory database.
Your only job is to convert the user's question into a single read-only
SQLite SELECT query, OR to ask a clarifying question when the request is
truly unresolvable.

DATABASE SCHEMA:
{schema_ddl}

RULES:
- Output ONLY a single JSON object. No prose, no markdown, no code fences.
- The JSON must be one of exactly these two shapes:
    {{"action": "query", "sql": "<a single SELECT statement>"}}
    {{"action": "clarify", "question": "<one short clarifying question>"}}
- Generate ONLY SELECT statements. Never INSERT, UPDATE, DELETE, DROP,
  ALTER, ATTACH, or PRAGMA.
- Use only the tables and columns defined in the schema above.

DEFAULT TO A FULL BREAKDOWN, DO NOT ASK UNNECESSARY QUESTIONS:
- When a user asks about a broad category (e.g. "how many tables do we
  have"), DEFAULT to returning the breakdown by subtype using GROUP BY,
  so the answer layer can show both the detail and the total. For example
  for tables: SELECT subtype, SUM(quantity) AS quantity FROM furniture_item
  JOIN furniture_category ON furniture_item.category_id = furniture_category.id
  WHERE furniture_category.name = 'table' GROUP BY subtype.
- Prefer answering over asking. Only use "clarify" when the question is so
  vague that no reasonable SELECT could answer it (e.g. "tell me about the
  furniture" with no measure implied — count? value? location?), or when a
  key term has no plausible mapping to the schema.
- If the user just names a category or subtype, assume they want the
  quantity breakdown.

CONVERSATION CONTEXT:
- Earlier turns may include your own prior clarifying question and the
  user's reply. Use them to resolve the current request.
```

Note for the implementer: `{schema_ddl}` is `.format`-substituted at load time. The doubled braces `{{ }}` in the examples are so `str.format` leaves them as literal single braces — keep them.

### 7.2 `ANSWER_COMPOSITION_PROMPT`

```
You are a helpful assistant that answers questions about a furniture
inventory. You are given the user's original question and the rows
returned by a database query. Write a short, clear, natural-language
answer based ONLY on those rows.

RULES:
- Base your answer strictly on the provided rows. Do not invent numbers.
- When the rows are a breakdown, state the breakdown AND the total.
  Example: "You have 18 tables in total: 10 square tables, 5 round
  tables, and 3 standing tables."
- If the rows are empty, say that no matching items were found.
- Do not output SQL. Do not mention the database or the query mechanics.
- You have no ability to run queries or take actions; only summarize the
  given rows.
```

The answer-composition call receives **only** the question and the result rows — never any capability to produce SQL — so that malicious text inside a data field cannot cause query execution.

## 8. LangGraph pipeline (`pipeline/graph.py`, `pipeline/nodes.py`)

### 8.1 State

```python
class State(TypedDict):
    question: str            # the current user message
    conversation: list       # prior (role, content) turns for clarify round-trips
    sql: str | None
    last_error: str | None   # validator feedback for retry
    attempts: int
    rows: list | None
    answer: str | None       # final natural-language answer
    clarify: str | None      # set if the graph is asking the user a question
```

### 8.2 Nodes

- **`generate_sql`** — calls the LLM with `SQL_GENERATION_PROMPT`. Builds the message list from `conversation` + the current `question`, and if `last_error` is set, appends a note: `"Your previous SQL was rejected: {last_error}. Produce a corrected SELECT."` Parses JSON. On `action=="clarify"`, sets `state.clarify`. On `action=="query"`, sets `state.sql`. Increment `attempts`. On JSON parse failure, set `last_error` and leave `sql`/`clarify` unset (treated as a retry).

- **`validate_sql`** — runs `validate_sql()` from `sql_guard`. On success, clears `last_error`. On failure, sets `last_error` to the reason.

- **`execute_query`** — calls `run_query()`; stores `rows`. Wrap in try/except; on DB error, set `last_error` and route back to `generate_sql`.

- **`compose_answer`** — calls the LLM with `ANSWER_COMPOSITION_PROMPT`, passing the question and `rows` (as JSON). Sets `answer`.

### 8.3 Edges

```
generate_sql --(clarify set)------------------> END
generate_sql --(sql set)----------------------> validate_sql
generate_sql --(neither, parse failed)--------> validate_sql retry path (see caps)

validate_sql --(valid)------------------------> execute_query
validate_sql --(invalid & attempts < MAX)-----> generate_sql
validate_sql --(invalid & attempts >= MAX)----> compose_answer   # graceful fallback

execute_query --(ok)--------------------------> compose_answer
execute_query --(db error & attempts < MAX)---> generate_sql

compose_answer -------------------------------> END
```

- `MAX_ATTEMPTS = 2`. When exhausted, `compose_answer` should produce a graceful "I couldn't build a valid query for that — could you rephrase?" message (pass an explicit empty/failure signal so it doesn't hallucinate data).
- Expose a single entrypoint, e.g. `def run_turn(question: str, conversation: list) -> dict:` that invokes the compiled graph and returns `{"answer" | "clarify", ...}` for the UI.

## 9. Chat UI (`chat.py`, terminal REPL)

A single `main()` loop, run with `python chat.py`. No web framework.

- On startup: verify `OPENROUTER_API_KEY` is set — if missing, print a clear message and exit non-zero. Print a one-line banner ("Furniture inventory assistant — ask a question, or type 'exit' to quit.").
- Maintain a `conversation` list (the same structure passed into the graph) across loop iterations so clarification round-trips work.
- Loop:
  1. Read a line with `input("you> ")`. Treat empty input as "ask again". Treat `exit`, `quit`, or Ctrl-D/Ctrl-C as a clean shutdown.
  2. Print a lightweight "…thinking" indicator (or just a newline) while `run_turn(question, conversation)` executes.
  3. If the result is a **clarify**, print the clarifying question prefixed like `bot> ` and append it to `conversation` so the user's next line resolves it.
  4. If the result is an **answer**, print it prefixed `bot> `.
  5. Continue.
- Wrap the LLM/DB call in try/except so a single failed turn prints an error and returns to the prompt instead of crashing the loop.
- Keep it plain `print`/`input`. Color or `rich` formatting is optional and must not add a required dependency.

## 11. Acceptance tests (verify these before finishing)

Provide a small `tests/` or a `test_manual.md` demonstrating each:

1. **Breakdown default** — "how many tables do we have?" → answer states 18 total with the 10/5/3 subtype breakdown. (No clarifying question asked.)
2. **Direct count** — "how many office chairs?" → 25.
3. **Total across everything** — "how many pieces of furniture in total?" → sum of all quantities.
4. **Clarify path** — "tell me about the furniture" → the system asks a clarifying question (e.g. count, value, or location?), and a follow-up like "the total value" then returns a real answer using the prior turn as context.
5. **Safety: write attempt is blocked** — force/verify that if the model ever emits `DELETE FROM furniture_item`, `validate_sql` rejects it and the DB is untouched. Add a direct unit test calling `validate_sql("DELETE FROM furniture_item", ALLOWED)` and asserting `is_valid is False`.
6. **Safety: schema probe blocked** — `validate_sql("SELECT * FROM sqlite_master", ALLOWED)` returns `False`.
7. **Safety: statement chaining blocked** — `validate_sql("SELECT 1; DROP TABLE furniture_item", ALLOWED)` returns `False`.
8. **Read-only connection** — attempting a write through the read-only connection in `pipeline/db.py` raises `sqlite3.OperationalError` (proves the second safety layer).

## 12. Guardrails for you, the implementer

- Do not add ORMs, Alembic, Docker, FastAPI, Streamlit, or a JS frontend. The interface is the terminal only.
- Do not weaken the two-layer safety model (validator + read-only connection). Both are required.
- Keep the two system prompts separate; never give the answer-composition call the ability to emit SQL.
- If any library API differs from what's described here, adapt the call but preserve the behavior and the safety guarantees.
- Print/log the generated SQL to the console on each turn (helpful for debugging), but never surface raw SQL in the chat UI.
